Code Review个人清单

git commit

• commit message遵循 angular社区规范
• 不包含其他feature分支的commit记录，不包含本功能之外的修改
• 提交记录保持开发一个功能对应一个commit
• 提交功能pr的时候 不要包含格式化的修改（单独提交）

代码风格

• 函数变量命名规范 复杂逻辑判断条件抽出常量命名
• 注释去解释特殊的产品逻辑和复杂的interface定义
• 代码不要出现“魔数” 特殊常量走配置化
• utils参数多补充注释 代码尽量不要有副作用 可测试性

React和typescript

• ts中的any控制在极个别的情况
• ts枚举尽量用英文 复杂或专业产品的名词可以用中文
• hooks依赖尽量做到依赖诚实 向下传参的时候注意函数和对象尽量不要内联写法（常常会造成死循环）
• 样式的逻辑切换 封装在styled-component里

代码bad smell

(根据社区一致的认可 删减一些)
• 重复的逻辑代码，样式重复。
• 巨大的组件
• 过长的方法与函数
• 过长参数列
• 发散式变化，霰弹式修改
• 人为的复杂: 在简单设计已经满足需求的时候，强迫使用极度复杂的设计模式。
• if逻辑复杂 及时重构历史债务 编写高质量可维护的代码：逻辑判断
